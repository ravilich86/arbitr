# Межбиржевой фьючерсный арбитраж (USDT-перпы)

Скрипт ищет межбиржевые расхождения цен по одной и той же бессрочной фьючерсной паре
(USDT-перпы) на биржах — **Binance, Bybit, OKX, MEXC, Bitget, Gate** — и открывает
дельта-нейтральную пару сделок (шорт на дорогой бирже, лонг на дешёвой), удерживает
её до схождения цен и закрывает с прибылью.

> Реализация по [ТЗ](TZ_futures_arbitrage.md). Только линейные перпы `*/USDT`.
> Никаких прогнозов направления рынка — только межбиржевой арбитраж одного актива.

## Ключи API

Секреты держатся **только** в `.env` (в `.gitignore`), в `config.yaml` — лишь имена
переменных. Права ключей на старте: **только торговля фьючерсами, без вывода средств**.
Где поддерживается — привязать ключ к IP сервера. OKX и Bitget требуют passphrase
(`OKX_API_PASSWORD`, `BITGET_API_PASSWORD`).

> **MEXC:** отдельно проверь, что у аккаунта включён доступ к API фьючерсов на
> **выставление ордеров** — MEXC для части аккаунтов ограничивает торговый
> futures-API (маркет-дата и funding обычно доступны, а `create_order` может быть
> закрыт). В `dry_run` это неважно, но перед боевым запуском протестируй один ордер.

## Конфигурация

Все параметры — в [`config.yaml`](config.yaml) (комментарий над каждым параметром).
Ниже полный справочник по секциям.

### Режим запуска

- `dry_run` — `true`: реальные ордера не отправляются, исполнение симулируется
  (paper), логируются сигналы и P&L. `false`: боевой режим с реальными деньгами.
- `testnet` — использовать testnet бирж, где поддерживается.

### `ws` — WebSocket-стриминг котировок

- `method` — способ стрима: `auto` (watch_bids_asks → tickers → стакан) либо
  принудительно `bids_asks` / `tickers` / `order_book`. Батчевый BBO снижает число
  подписок и убирает ошибки checksum и лимит `request too many`.
- `symbols_per_stream` — размер чанка подписки (для режима стакана).
- `subscribe_delay` — пауза (сек) между стартами чанков подписок.
- `warmup` — сек прогрева перед первым сканом.
- `funding_interval` — сек между обновлениями funding по всем парам.
- `stats_interval` — сек между строками диагностики (покрытие + топ спреды).

### `exchanges` — биржи

По каждой: `enabled`, `api_key_env` / `api_secret_env` (+ `api_password_env` для
OKX и Bitget), `taker_fee` (доля), `default_type: swap`.

### `fees` — комиссии

- `fetch_from_exchange` — `true`: на старте бот тянет **актуальные taker-комиссии
  с бирж** (`fetch_trading_fees`, учитывает VIP-уровень аккаунта) и использует их
  в расчётах вместо конфига. При неудаче для конкретной биржи берётся её
  `taker_fee` из секции `exchanges`. `false` — всегда значения из конфига.

### `spread` — пороги спреда и выхода

- `min_gross_spread` — **базовый** порог входа «сейчас разошлось» (сырой спред).
- `max_gross_spread` — верхняя граница (выше = ошибка данных).
- `min_net_spread` — порог чистой прибыли после издержек.
- `exit_spread` — целевое схождение для закрытия.
- `take_profit` — закрыть при достижении P&L (USDT); `null` = выкл.
- `min_spread_persistence` — сек удержания спреда до входа (отсекает флэш).
- `max_quote_age_ms` — макс. возраст котировки.
- `hunt_flash` — охота за прострелами (выкл).

### `history` — историческая сверка тождественности

- `enabled`, `prefilter` (сверка один раз на старте vs ленивая), `prefilter_concurrency`.
- `require_divergence` — сверка обязательна для всех входов.
- `check_spread` — с какого текущего спреда считать «разошлось».
- `timeframe`, `days` — свечи и период.
- `max_divergence` — **базовый** порог «исторически шли вместе» (0.3%).
- `max_candidates_check` — сколько верхних сигналов проверять историей (ленивый режим).

### `filters` — вкл/выкл доп-фильтров сигнала

Каждый включается отдельно: `net_spread`, `top_depth`, `persistence`,
`max_gross_spread`, `quote_age`. Базовые критерии (история ≤0.3% + сейчас ≥1%)
работают всегда. Отключай лишнее, если сигналов мало.

### `sizing` — объём и плечо

- `notional_target` — целевой нотионал одной ноги (USDT). Задаёт объём:
  `объём = notional / цена`, округлённый под шаг лота. Влияет на исполнимость,
  цену входа (глубже по стакану — хуже средняя) и P&L.
- `position_size` — явный нотионал; если задан, используется **вместо**
  `notional_target`. `null` = использовать `notional_target`.
- `leverage` — желаемое плечо (факт = min(leverage, макс. плечо биржи)).
- `margin_mode` — `isolated` | `cross`.
- `max_slippage` — порог фильтра и слагаемое в *оценке* чистого спреда; в
  фактический P&L не входит (там слиппедж по реальному стакану).

### `execution` — исполнение

- `order_type` — `market` | `limit`.
- `on_leg_failure` — `rollback` | `retry` при незахеджированной ноге.
- `leg_timeout` — сек ожидания исполнения ноги.
- `simulate_slippage` — dry_run: исполнять по VWAP реального стакана.
- `orderbook_limit` — глубина стакана для оценки исполнения.

### `risk` — управление рисками

- `max_concurrent_positions` — макс. одновременных пар; **0 = без лимита** (берём по
  максимуму, пока хватает баланса).
- `max_position_per_exchange` (USDT залога на бирже), `max_hold_time` (сек),
  `max_adverse_spread`, `liquidation_buffer`, `cooldown` (сек).
- `profit_buffer_close` — закрывать убыточные пары за счёт накопленной прибыли
  (см. ниже); `profit_buffer_keep` — минимальный сессионный net, ниже которого не опускаемся.

### Неттинг убытков прибылью (`profit_buffer_close`)

Если по паре нереализованный убыток, а сессионный P&L его перекрывает, бот
закрывает убыточную позицию сейчас, «профинансировав» её накопленной прибылью
(начиная с самых убыточных). Правило безопасности: после закрытия сессионный net
не опускается ниже `profit_buffer_keep`. Это ограничивает удержание расходящихся
позиций. По умолчанию выключено (реализует убыток раньше срока — включай осознанно).

### Баланс и лимиты

В боевом режиме перед входом бот читает **реальный свободный баланс** биржи
(`fetch_balance`); если маржи не хватает — вход пропускается и в Telegram уходит
алерт (пункт 8). После закрытия любой позиции логируется и добавляется в пуш
**общий баланс всех бирж** (пункт 10).

### `universe` — вселенная / коллизии

- `skip_delisting_days` — пропускать пары с делистингом в пределах N дней.
- `max_contract_size_ratio` — `null` = проверка по contractSize выключена (см. конфиг).

### `allow_list` / `deny_list`, `logging`, `telegram`

- `allow_list` — если непусто, отслеживать только эти тикеры; `deny_list` — исключить.
- `logging` — `app_log`, `trades_log` (jsonl|csv), `level`.
- `telegram` — `enabled`, `token_env`, `chat_id_env`, `app_name` (метка в сообщениях).

### Как считается P&L (dry_run)

Объём ноги = `notional_target` ÷ цена, округлённый под шаг лота. Цена входа/выхода —
**VWAP по реальному стакану** для этого объёма (`execution.simulate_slippage: true`):
бот проходит по уровням, шорт наливается в bids, лонг — в asks; при нехватке глубины
исполнение частичное. Формула:

```
P&L = (цена_шорта_вход − цена_шорта_выход) × объём
    + (цена_лонга_выход − цена_лонга_вход) × объём
    − комиссии_входа − комиссии_выхода           (taker_fee обеих ног ×2)
    + funding_за_удержание                        (funding_H − funding_L, оценка)
```

Это **консервативная оценка**, а не идеальная и не гарантированная прибыль:
не моделируются задержка между ногами и мгновенное изменение стакана, а комиссии
берутся из `taker_fee` конфига (выставь реальные). При `simulate_slippage: false`
берётся цена верхушки стакана (оптимистично). «Чистый спред» в уведомлении о входе —
это *оценка на момент сигнала*, а «💰 P&L» при закрытии — *факт* по VWAP-исполнению.

## Запуск

```bash
python -m arb.main --config config.yaml                # цикл на WebSocket (Ctrl+C для остановки)
python -m arb.main --config config.yaml --iterations 5 # ограниченный прогон
python -m arb.main --config config.yaml --rest         # REST вместо WebSocket (медленно)
```

По умолчанию работает на **WebSocket**: фоновые стримы непрерывно обновляют кэш
котировок, а цикл сканирует кэш — это на порядок быстрее REST. Для котировок
используется **батчевый BBO** (`watch_bids_asks`): один поток отдаёт лучший
bid/ask сразу по группе пар (`ws.symbols_per_stream`), а не отдельный стакан на
каждую пару. Это кратно снижает число подписок и убирает ошибки checksum стакана
и лимит `request too many`. Метод настраивается (`ws.method`: auto/bids_asks/
tickers/order_book). Флаг `--rest` оставлен для отладки.

Логи: технический — `logs/app.log`, сделки — `logs/trades.jsonl` (одна сделка = одна
запись со всеми числами и итоговым P&L). Сводка сессии печатается по завершении.

## Аварийное управление позициями

На случай форс-мажора:

```bash
python -m arb.main --close-all    # ПАНИКА: закрыть ВСЕ позиции на всех биржах
python -m arb.main --reconcile    # проверить парность и закрыть незахеджированных орфанов
```

Обе команды читают **реальные** позиции с бирж (`fetch_positions`), поэтому работают
независимо от состояния бота. `--reconcile` спаривает ноги в дельта-нейтральные
арб-пары (шорт на одной бирже + лонг на другой, сопоставимый объём) и закрывает
только «орфанов» — одиночные ноги без противоположной пары (наведённый
направленный риск). `--close-all` закрывает всё подряд.

Защита от случайного срабатывания: при `dry_run: true` команды выдают только
**отчёт** (что было бы закрыто); чтобы реально отправить ордера — добавь `--force`
или запускай в боевом режиме (`dry_run: false`).

**Проверка на каждом старте.** При обычном запуске бот сам вызывает реконсиляцию:
читает позиции с бирж, ищет орфанов и при обнаружении шлёт алерт (лог + Telegram).
Автозакрытие орфанов на старте — по флагу `risk.close_orphans_on_start` (только в
боевом режиме).

**Локальное хранилище** (`state`): бот ведёт JSON-файл открытых позиций
(`data/positions.json`) для аудита. Для аварийного закрытия оно **не требуется** —
позиции читаются прямо с бирж; файл нужен лишь как журнал намеренных пар.

## Уведомления в Telegram

Бот шлёт пуш при старте, при входе в арбитраж и при закрытии позиции. Токен и
chat_id — в `.env` (`TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`), настройки — в
`config.yaml → telegram`. Каждое сообщение помечено `app_name`, чтобы отличать его
от статистики других ботов в том же чате.

**Политика в боевом режиме** (`live_only_trades: true`, пункт 9): при `dry_run: false`
в Telegram уходят пуши по открытию/закрытию позиций **и критичные алерты о нехватке
баланса** (баланс — исключение, шлётся всегда); старт, аномалии и диагностика — нет.
В `dry_run` шлётся всё. Проверить связь:

```bash
python -m arb.main --test-telegram
```

## Защита от фейковых сигналов

Кросс-биржевые «спреды» в 5–10% почти всегда — не арбитраж, а артефакт данных
(протухшая котировка, пустой стакан, разные единицы контракта). Отсекается так:

- Коллизии (один тикер = разные токены) проявляются в ЦЕНЕ и отсекаются
  `spread.max_gross_spread` (см. ниже) + `deny_list`. Проверка по `contractSize`
  выключена по умолчанию (`universe.max_contract_size_ratio: null`): множитель
  контракта законно отличается между биржами на порядки (MEXC = 0.0001, Binance = 1)
  и не влияет на цену — ccxt нормализует котировки к цене за 1 базовую единицу.
- `spread.max_gross_spread` — верхняя граница сырого спреда; выше считаем ошибкой данных.
- `spread.max_quote_age_ms` — котировки старше отбрасываются как устаревшие.
- Грубая проверка глубины верхушки стакана — что наш объём пролезает на вход И на выход.

Все «тяжёлые» проверки (чистый спред, глубина, устойчивость, верхняя граница спреда,
устаревание котировок) включаются/выключаются в `config.yaml → filters`. Базовые
критерии действуют всегда: (1) исторически шли вместе ≤0.3% и (2) сейчас разошлись
≥1%. Если сигналов слишком мало — можно отключить лишние фильтры. Причина
отклонения топ-спредов видна в строках `Диагностика:`.
- **Историческая сверка «было вместе → разошлось»** (`history`): вход только если
  пара сейчас разошлась на `check_spread` (≥1%) И исторически шла вместе — за
  `days` дней (10) на дневных свечах её min/max на биржах расходятся не более чем на
  `max_divergence` (0.3%). При `prefilter: true` (по умолчанию) историческая сверка
  выполняется **один раз на старте** (`prequalify_universe`): анализируются
  10-дневные свечи всех пар, и в работе остаются только те, что исторически шли
  вместе; в момент сигнала свечи повторно не тянутся. При `prefilter: false` —
  ленивая сверка в момент сигнала. Кандидаты перебираются по убыванию net.
- **Пропуск делистинга** (`universe.skip_delisting_days`): пары, у которых на
  какой-то бирже анонсирован делистинг/поставка в пределах N дней, исключаются —
  арбитраж по исчезающей ноге небезопасен.
