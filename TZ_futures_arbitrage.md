# Техническое задание: скрипт фьючерсного арбитража между биржами

## 0. Кратко о задаче

Нужен скрипт, который автоматически ищет **межбиржевые расхождения цен** по одной и той же бессрочной фьючерсной паре (USDT-перпы) на пяти биржах и открывает дельта-нейтральную пару сделок (шорт на дорогой бирже, лонг на дешёвой), удерживает её до схождения цен и закрывает с прибылью.

**Биржи:** Binance, Bybit, OKX, Gate, Bitget.
**Тип инструмента:** только линейные бессрочные фьючерсы (USDT-маржинальные перпы, `*/USDT`).
**Режим на старте:** реальные деньги, по 100 USDT на каждой бирже (бюджет 500 USDT суммарно), минимальный объём сделки, плечо до 20x.

> Это арбитраж только между биржами по одному и тому же активу. Никаких прогнозов направления рынка. Низколиквидные пары не исключаются — наоборот, расхождения ≥0.5% живут именно на них; единственное требование к паре: она торгуется минимум на 2 из 5 бирж, и через её стакан проходит **наш** объём на вход и на выход.

---

## 1. Стек и инфраструктура

- Язык: **Python 3.11+** (рекомендую из-за зрелых async-библиотек для бирж).
- Работа с биржами: библиотека **CCXT** (`ccxt.pro` для WebSocket-стримов) — даёт единый интерфейс ко всем пяти биржам, поддерживает futures/swap и funding rate. Если CCXT где-то не покрывает нужный метод — точечно использовать нативный REST/WS этой биржи.
- Асинхронность: `asyncio` — все 5 бирж опрашиваются параллельно, ордера на двух ногах выставляются конкурентно.
- Хранение конфигурации: отдельный файл `config.yaml` (ключи API, параметры стратегии) + `.env` для секретов. Ключи **никогда** не хардкодить в коде.
- Запуск на выделенном сервере (есть). Желательно располагать сервер ближе к дата-центрам бирж и использовать постоянные WebSocket-соединения, чтобы минимизировать задержку.
- Все API-ключи на старте — с правами **только на торговлю фьючерсами**, **без права на вывод средств**. Если биржа поддерживает IP-whitelist — привязать ключи к IP сервера.

---

## 2. Модули скрипта

1. `exchanges` — инициализация коннекторов ко всем 5 биржам, загрузка рынков, обработка лимитов запросов (rate limits).
2. `universe` — сбор всех USDT-перпов и построение списка **тождественных пар** (см. §3–4).
3. `marketdata` — поток котировок (лучший bid/ask) и funding rate по отслеживаемым парам.
4. `scanner` — поиск расхождений ≥ порога с учётом комиссий и funding (см. §5–6).
5. `executor` — открытие/закрытие двух ног, контроль частичного исполнения, откат (см. §7–8).
6. `risk` — контроль плеча, ликвидации, лимитов на позицию и одновременные сделки (см. §9).
7. `logger` — структурированные логи всех событий и сделок (см. §10).

## 3. Этап 1 — сбор всех USDT-фьючерсных пар

Для каждой из 5 бирж:
- Загрузить список рынков (`load_markets`).
- Отфильтровать только: тип `swap` (бессрочный фьючерс), линейный (`linear`), котируемый в USDT, статус «активен/торгуется».
- Сохранить по каждому символу: нормализованный базовый актив, точный биржевой тикер, размер тика цены (tick size), шаг количества (step size / lot size), минимальный объём (min qty / min notional), максимально доступное плечо, частоту и время начисления funding.

Результат этапа — для каждой биржи словарь «нормализованный символ → метаданные контракта».

---

## 4. Этап 2 — определение тождественных пар

Цель: получить пары вида «один и тот же актив, торгуется минимум на 2 из 5 бирж».

- Нормализовать тикеры: убрать различия в записи (`BTC/USDT:USDT`, `BTCUSDT`, `BTC-USDT-SWAP` → единый ключ `BTC/USDT`).
- **Важно — защита от ложных совпадений:** один и тот же тикер на разных биржах может означать разные токены (коллизии тикеров). Прежде чем считать пару тождественной, желательно сверять по дополнительному признаку (например, contract size / underlying, либо вести ручной allow-list / deny-list для спорных тикеров в конфиге). Лучше пропустить сомнительную пару, чем открыть арбитраж между двумя разными активами.
- Сформировать матрицу: для каждого актива — на каких биржах он есть и с какими метаданными.
- Отслеживать только активы, присутствующие минимум на 2 биржах. Это и есть пул кандидатов для арбитража.

---

## 5. Этап 3 — учёт ставок финансирования (funding rate)

Так как позиция дельта-нейтральная (лонг на одной бирже + шорт на другой) и **удерживается во времени**, funding влияет на итоговый результат и должен учитываться явно.

Механика (исполнитель обязан перепроверить по докам каждой биржи, ниже — общая логика перпов):
- Положительный funding: лонги платят шортам. Отрицательный: шорты платят лонгам.
- Мы держим **шорт** на дорогой бирже H и **лонг** на дешёвой бирже L.
- Чистый funding за период ≈ `funding_H − funding_L` (× нотионал). Положительное значение = мы получаем, отрицательное = платим.

Требования:
- Для каждой кандидатной пары подтягивать текущий funding rate и время следующего начисления с обеих бирж.
- **Фильтр входа по funding:** входить только если ожидаемый чистый funding за планируемое время удержания **не съедает спред**. То есть условие прибыльности (см. §6) должно учитывать funding как издержку/доход.
- Если до ближайшего начисления funding осталось мало времени и оно работает против нас — либо не входить, либо закладывать его в расчёт.
- Параметры частоты funding (8ч / 4ч / иное) у бирж различаются — брать фактическое значение по контракту, не хардкодить 8 часов.

## 6. Этап 4 — поиск расхождений и условие прибыльности

**Сигнал входа:** по одному активу находим **пару бирж с максимальным расхождением** цены, где спред ≥ порога.

Расчёт спреда (по реально исполнимым ценам, а не по «средней» цене):
- Дешёвая нога (лонг): берём **ask** на бирже L (мы покупаем по ask).
- Дорогая нога (шорт): берём **bid** на бирже H (мы продаём по bid).
- Сырой спред = `(bid_H − ask_L) / ask_L`.

**Порог.** Заказчик задал минимум 0.5%. Но 0.5% — это *сырой* спред. Чистая прибыль = сырой спред − комиссии обеих ног (вход + выход) − чистый funding за удержание − ожидаемый слиппедж. Поэтому в конфиге задаём:
- `min_gross_spread` (по умолчанию 0.5%) — порог сырого расхождения;
- `min_net_spread` (по умолчанию, например, 0.15–0.25%) — порог **чистой** прибыли после всех издержек.

Скрипт открывает сделку **только если `net_spread ≥ min_net_spread`**. Это гарантирует, что условие «чтобы не было убытка и был хотя бы небольшой плюс» выполняется, а не просто визуально видно 0.5%.

Комиссии: для каждой биржи брать актуальные **taker-комиссии** фьючерсов из конфига (исполнитель должен заполнить реальными значениями — обычно это доли процента, но они отличаются по биржам и по уровню аккаунта/VIP). Round-trip = 4 исполнения (2 входа + 2 выхода).

**Проверка исполнимости объёма (не фильтр ликвидности!):** низколиквидные пары допустимы и ожидаемы. Проверяем не «глубокий ли стакан вообще», а узкий вопрос: пролезает ли **наш** ордер (~$2000 нотионала при $100×20x) на вход на обеих биржах с приемлемым слиппеджем, заложенным в `min_net_spread`. Дополнительно — **обязательно проверять и сторону выхода**: арбитраж = круговая сделка, и если на одной из бирж нельзя будет закрыть ногу без большого слиппеджа, сигнал лучше пропустить, иначе схождение случится, а зафиксировать прибыль не получится. Порог слиппеджа (`max_slippage`) — в конфиге; на тонких стаканах надёжнее лимит/post-only ордера, чем market.

**Тип расхождения — устойчивое vs флэш (важно для leg-риска).** Гэп между биржами возникает от дисбаланса крупных заявок на тонком стакане (крупный покупатель на одной бирже, продавец на другой). Различаем два режима:
- *Устойчивое расхождение* — одна биржа отстаёт, гэп держится секунды/минуты (классический пример: токен «гуляет» в диапазоне между биржами). Обе ноги успеваешь налить спокойно. **Это основная цель на старте.**
- *Флэш-прострел* — гэп от мгновенного всплеска, схлопывается за секунды. Гнаться за ним маркет-ордерами на двух биржах одновременно — максимум leg-риска (вторая нога не успевает). На старте с $500 такой режим по умолчанию выключен.

Параметр `min_spread_persistence` — сколько времени расхождение должно продержаться ≥ порога, прежде чем скрипт войдёт (отсекает флэш-прострелы). Включение охоты за прострелами — отдельным флагом, осознанно.

**Сканирование:** по всем кандидатным парам, по всем сочетаниям бирж, в реальном времени (WebSocket лучше, чем частый REST-polling, из-за свежести данных и лимитов). Выбираем сочетание с максимальным net-спредом, проходящее фильтры.

## 7. Этап 5 — открытие сделок (две ноги)

Когда сигнал прошёл все фильтры:
- На бирже **H (цена выше)** открыть **SHORT**.
- На бирже **L (цена ниже)** открыть **LONG**.
- Количество базового актива на обеих ногах должно быть **равным** (дельта-нейтральность), с округлением под `step size` каждой биржи — берём наибольшее общее допустимое количество, удовлетворяющее минимумам обеих бирж.

**Критично — синхронность и leg-risk:**
- Ноги выставлять **конкурентно** (параллельно), чтобы не упустить спред.
- Если одна нога исполнилась, а вторая нет (отклонена / частично / таймаут) — это незахеджированная позиция. Нужна логика отката: либо немедленно довыставить вторую ногу, либо закрыть уже открытую ногу по рынку и зафиксировать сигнал как несостоявшийся. Поведение задаётся в конфиге (`on_leg_failure: rollback | retry`).
- Обрабатывать **частичное исполнение**: довести обе ноги до одинакового размера или откатить до симметрии.
- Тип ордера на вход: настраиваемый. Маркет — быстрее, но больше слиппедж; лимит/post-only — дешевле, но риск неисполнения. На старте для надёжности входа допустимо market, но слиппедж закладывать в `min_net_spread`.

**Выход (заказчик описал «ждём приближения» — формализуем):**
- Цель закрытия: когда спред между ногами сходится до `exit_spread` (по умолчанию около 0% или небольшой положительный таргет) — закрыть обе ноги одновременно (шорт → выкупаем, лонг → продаём).
- `take_profit` (опционально): зафиксировать, когда чистая прибыль по позиции достигла целевого значения.
- **Защита от расхождения (обязательно):** спред может не сойтись, а разойтись ещё сильнее. Нужны:
  - `max_hold_time` — максимальное время удержания, после которого позиция закрывается принудительно (даже в небольшой минус), чтобы не копить funding и риск.
  - `max_adverse_spread` — если спред разошёлся сверх лимита, принять решение по конфигу (закрыть/удерживать/добрать). По умолчанию — контролируемое закрытие, без «надежды».
- Выход тоже выставлять конкурентно по обеим ногам, с такой же обработкой leg-risk, как при входе.

---

## 8. Этап 6 — объём и плечо

- Объём на старте — **минимально возможный**, одинаковый на обеих биржах в единицах базового актива, с учётом step size и min notional каждой биржи.
- Плечо: **20x**, а где 20x недоступно — **максимально возможное до 20x** (берём `min(20, max_leverage_биржи)`).
- Тип маржи: рекомендуется **isolated** на старте, чтобы риск по одной паре не затронул весь баланс биржи. Параметр в конфиге (`margin_mode: isolated | cross`).
- На каждой бирже одновременно может быть открыта только согласованная нога нужного размера; контроль — в модуле risk.

## 9. Управление рисками (модуль risk)

Это часть ТЗ, потому что напрямую влияет на то, будет ли стратегия в плюсе, а не только в теории.

- **Ликвидация при 20x.** При 20x ликвидация одной ноги наступает примерно при движении против неё на ~5% (зависит от биржи и поддерживающей маржи). В арбитраже одна нога всегда движется против нас. Маржа изолирована по биржам, мгновенно перебросить её нельзя. Поэтому:
  - расстояние до ликвидации каждой ноги должно быть заметно больше ожидаемого максимального расхождения спреда;
  - параметр `liquidation_buffer` — минимальный запас до цены ликвидации; при его нарушении — алерт и/или сокращение/закрытие позиции;
  - при необходимости снижать эффективное плечо позиции (меньше нотионал на тот же залог), даже если биржа разрешает 20x.
- `max_concurrent_positions` — лимит одновременных арбитражных сделок (на старте, например, 1).
- `max_position_per_exchange` — чтобы не превысить 100 USDT залога на бирже.
- `cooldown` — пауза по паре после закрытия, чтобы не дёргать один и тот же сигнал.
- Контроль балансов: перед входом проверять, что на обеих биржах хватает свободной маржи.
- Обработка ошибок API, разрывов WebSocket, рассинхрона времени (особенно важно для funding и подписи запросов — синхронизировать NTP на сервере).
- **Kill-switch:** одна команда/флаг, по которой скрипт прекращает открывать новые сделки и (опционально) закрывает все текущие.

## 10. Логирование

Вести два потока логов:

1. **Технический лог** (`app.log`): запуск, подключения, ошибки, разрывы, rate-limit, отклонённые сигналы (с причиной — спред < порога / funding / нет глубины / leg-fail).
2. **Лог сделок** (`trades.csv` или JSONL, машиночитаемо + человекочитаемо): по каждой арбитражной сделке отдельной записью:
   - id сделки, время входа/выхода;
   - актив, биржа-H (шорт) и биржа-L (лонг);
   - сырой спред на входе, чистый ожидаемый спред, фактический спред на выходе;
   - объём (база и нотионал), плечо;
   - цены и комиссии входа/выхода по каждой ноге;
   - начисленный funding за время удержания;
   - **итоговый P&L по сделке (в USDT и в %)** и причина закрытия (target / take-profit / max_hold_time / max_adverse / kill-switch);
   - leg-status (обе ноги ок / был откат / частичное).
- Дополнительно — сводка за сессию: число сделок, винрейт, суммарный P&L, средний P&L на сделку, число пропущенных сигналов.

Логи должны читаться без разбора кода: одна сделка = одна понятная строка/запись со всеми числами.

## 11. Конфигурация (`config.yaml`) — что вынести в настройки

- Списки API-ключей по биржам (через `.env`), флаги вкл/выкл каждой биржи.
- `min_gross_spread`, `min_net_spread`, `exit_spread`, `take_profit`.
- `taker_fees` по каждой бирже.
- `leverage` (20), `margin_mode`, `order_type` (market/limit), `on_leg_failure`.
- `position_size` / минимальный объём, `max_concurrent_positions`, `max_position_per_exchange`.
- `max_hold_time`, `max_adverse_spread`, `liquidation_buffer`, `cooldown`.
- `allow_list` / `deny_list` тикеров.
- `dry_run` — режим без реальных ордеров (только лог сигналов): **сильно рекомендую прогнать на нём перед запуском реальных денег**, чтобы убедиться, что сигналы и расчёт P&L корректны. Многие из этих бирж дают testnet — на нём же отладить логику исполнения и откатов до боевого режима.

## 12. Порядок реализации (предлагаемый)

1. Коннекторы + сбор USDT-перпов (§3).
2. Матрица тождественных пар (§4).
3. Поток котировок + funding (§5).
4. Сканер с расчётом net-спреда и фильтрами (§6) — сначала в `dry_run`, только логирование сигналов.
5. Исполнитель: вход двумя ногами, leg-risk, откат (§7).
6. Логика выхода и риск-контроль (§7–9).
7. Логирование и сводки (§10).
8. Прогон на testnet / dry_run → затем боевой запуск с минимальным объёмом.

## 13. Допущения и что вне рамок (на старте)

- Только USDT-перпы, только 5 указанных бирж.
- Без вывода/перевода средств между биржами автоматически (ключи без права вывода).
- Без сложных портфелей из 3–4 инструментов — пока классические две ноги (это можно добавить позже).
- Реальные значения комиссий, минимальных объёмов и доступного плеча по каждой бирже исполнитель должен подставить из актуальной документации/аккаунта на момент запуска (они меняются).
